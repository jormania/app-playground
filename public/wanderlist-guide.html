Wanderlist — guide

A friendlier doorway to your Notion list of city things to do. Browse the backlog, triage it, mark things attended, and get a heads-up the day before a deadline closes.

What it is

Wanderlist reads and writes a single Notion database of things to do — events, culture, art, movies, venues, ideas. It’s a triaged backlog: keep everything you mean to get to in one place, sort by what’s expiring soonest, check items off as Attended without losing them — and see it all on a month calendar that plots each item’s planned date and expiring deadline.

Everything is stored in your Notion. The app keeps a local copy so it works offline and installs as an app; nothing lives on a server except the optional Place-search key and reminder setup below.

1 · Connect Notion

Wanderlist reads and writes a single Notion database. Until you connect one, the app stays in demo mode on sample items stored only on this device. To keep real entries you need two things: a Notion database with the right columns, and an integration token that lets the app read and write it. Setting this up takes a few minutes and needs no coding.

Option A — duplicate the ready-made Starter Template (easiest)

Duplicate the Wanderlist Starter Template — an empty database with the exact columns already set up. Open it and use Duplicate (top-right ••• → Duplicate) to make your own private copy — then skip straight to section 2 to connect it.

Option B — build an empty one by hand

In Notion, create a new page and add a Table database (full-page is tidiest). Name it whatever you like. Then set up exactly these columns — the names matter, because the app looks them up by name:

ColumnTypeWhat it holds
NameTitleThe default first column. What it is, a few words.
DescriptionTextNotes, programme, why it caught your eye. Optional.
LinkURLThe event/listing page. Optional.
CategorySelectLeave the options empty, or seed event, discovery, venue, idea, culture, movie, art — the app adds more inline either way.
PlaceTextHolds the resolved place name. Not Notion’s native Maps/geo property — plain text.
MapURLThe Maps pin link for the place. Leave empty — the app fills it in.
TagsMulti-selectLeave the options empty — they fill in as you add things.
AttendedCheckboxDefaults unchecked.
GoingCheckboxDefaults unchecked. Separate from Attended — “have I decided to go”, not “did it happen”. Only shown in the editor once a Planned Date is picked.
Date AddedDateThe app stamps this on create.
Date ExpiringDateThe deadline to act on — a registration or ticket window closing. Optional.
Planned DateDateThe day you plan to go. Optional — and can optionally carry a start time too (no end time), if it's something with a fixed start.
PhotoFiles & mediaOptional. Leave empty — the app uploads into it. At most one file is ever used.
TicketsFiles & mediaOptional. Leave empty — the app uploads into it. Any number of files.

That’s every column. You can delete any extra columns Notion added by default as long as the thirteen above exist with these exact names. Category and Tags are always saved lowercase, whatever case you type, so they never fork into look-alike duplicates.

2 · Create the integration & connect

An internal integration is just a key that lets this one app reach your database. Here’s the whole path, step by step:

  1. Create the integration. Go to notion.so/my-integrations and click New integration. Give it a name (e.g. Wanderlist), pick the workspace your database lives in, and leave the type as Internal. Save it.
  2. Copy the token. On the integration’s page, open Configuration (or Secrets) and find Internal Integration Secret. Click Show, then Copy. It starts with ntn_ (older ones start with secret_). Treat it like a password.
  3. Share the database with it. Open your Wanderlist database in Notion. Top-right ••• menu → Connections (some workspaces label it Add connections) → search your integration’s name → select it. This is what grants read/write access — without it the app sees nothing.
  4. Get the database link. Still on the database, ••• → Copy link (or just copy the URL from your browser’s address bar while viewing it).
  5. Connect in the app. Open Wanderlist, tap the ⋯ menu → Settings, paste the token into the first field and the database link into the second, hit Test connection, then Save. The app flips from demo to your live backlog immediately.

Both the token and the database link are stored only in this browser and relayed to Notion through this site’s proxy — never saved on a server. Anyone using this browser profile can reach your backlog, so disconnect from Settings (which clears the token) on shared devices. To switch backlogs later, just paste a different database link.

3 · Place search optional

When you type in the Place field, Wanderlist can suggest real places (Google) and store the resolved name plus a Maps link — pick a suggestion once and it sticks; reopening the entry later won’t re-search it out from under you. This step is entirely server-side, done once by whoever runs the deployment — not something each person connecting their own Notion database needs to repeat.

  1. Open Google Cloud Console. Go to console.cloud.google.com and create a project (or pick an existing one) — top-left project selector → New Project.
  2. Enable Places API (New). In the console’s search bar, find Places API (New) — not the older “Places API” — under APIs & Services → Library, and click Enable.
  3. Create an API key. APIs & Services → Credentials → Create Credentials → API key. Copy the key.
  4. Restrict the key. Click into the key’s settings: set API restrictions to just Places API (New). Leave Application restrictions as None — the key is only ever called server-side (this site’s own proxy, never the browser), so a Websites/IP restriction would block the very requests it’s meant to allow.
  5. Add it to Vercel. In the project’s Vercel dashboard → Settings → Environment Variables, add:
GOOGLE_PLACES_KEY = your-google-places-key

Redeploy for the variable to take effect. Without the key — or if it’s missing or invalid — the Place field simply becomes a plain text box: you type the place name yourself and everything else (Category, Tags, dates, the calendar, reminders) still works exactly the same.

4 · Email reminder optional

Get one email around 7pm the evening before — for anything unattended that's Expiring, Planned, or Planned and Going the next day, each line labelled with which it is (an item can be more than one, e.g. expiring and planned the same day). It’s sent by a daily server job, so it reaches you even when the app is closed. Like Place search, this is a one-time server-side setup done once by whoever runs the deployment; after that, turning it on and setting the address is done per-person in Settings → Email reminder.

  1. Create a Resend account. Sign up at resend.com (the free tier is plenty for one person's reminders) and grab an API key from the dashboard.
  2. Pick a sender. The quickest path is Resend's own onboarding@resend.dev address, which can send to your own verified email without any domain setup. To send from your own address instead, verify your domain in Resend first.
  3. Create a Vercel KV store. In the Vercel project → Storage → KV → Create. This holds the reminder preferences (on/off, email, name) so the daily job can read them — connecting it auto-adds the needed KV_REST_API_* environment variables for you.
  4. Add the remaining environment variables in Vercel → Settings → Environment Variables:
WANDERLIST_NOTION_TOKEN = ntn_… (integration token, shared with the DB) WANDERLIST_DB_ID = your database id RESEND_API_KEY = re_… REMINDER_FROM = Wanderlist <onboarding@resend.dev> CRON_SECRET = any-long-random-string (optional, guards the endpoint)
  1. Redeploy. The job itself is already wired in vercel.json as two daily entries — one per Bucharest DST offset, so it lands around 7pm local year-round without needing a timezone-aware cron — nothing to add there. Once the vars are set and the project redeploys, open Settings → Email reminder in the app, turn it on, enter your email, and Save.
  2. Verify it actually works. Hit Send test reminder, right next to Save — it fires a real email immediately through the same pipeline the daily job uses (a placeholder item if nothing’s genuinely due tomorrow), so you can confirm delivery without waiting for a real deadline.

The reminder address and name are editable in the app (saved to KV, gated by your Notion token so only you can change them). Secrets stay in Vercel env vars and never touch your browser.

5 · Everyday use

  • Add anything with just a name — flesh it out later.
  • Filter with the To-do / Attended / All segment; search by text, category, place, or tags; tap any chip to filter by it. Your last view, filter, and sort stick across reloads.
  • Sort by expiring-first (default), recently added, or A–Z. Expiring-soon items float up with a warm pill; anything already past its deadline sinks below a past divider instead of jumping the queue.
  • Hit the round ✓ to mark something attended — it stays in the list, just checked off.
  • Planned Date can optionally carry a start time too, once a date's picked — no end time, just when it begins. Shows up everywhere the planned date does (list, detail, calendar agenda). Expires never has a time — it's a deadline day, not a moment.
  • Once a Planned Date is set, a Going toggle appears in the editor — off by default (“still deciding”), tap it once you've actually committed. It's separate from Attended, so you can track a concert's date and still be undecided about going. Confirmed items show a filled dot on the calendar and a green “going” pill instead of “planned”.
  • Photo — add one picture per item (a poster, a shot you took) from the editor; tap it in the detail view for the full size.
  • Tickets — attach one or more PDFs or screenshots, kept separate from Photo; each opens in a new tab from the detail view. Anything with tickets on file reads as paid — a gold badge in the list, and a gold dot/chip on the calendar. Tapping Paid with more than one ticket lists them so you can pick which to open; with exactly one, it opens directly.
  • Every list row has a quick-check rail on the right: Photo/Link/Paid show only when there's something there, and Going (once a Planned Date is set) and Attended are always-visible tap toggles — a fast way to update either without opening the item. Going here is the same field as the editor's checkbox, just a quicker path to it. Each gets its own colour used everywhere it shows up: green = Attended, blue = Going (hollow while undecided, filled once confirmed), gold = Paid — with a short vibration on tap where your device supports it.
  • Calendar (the calendar button): a month grid where each day shows up to three dots — Planned (hollow until you've marked something Going, then filled solid), Expiring, and Paid. Tap a day to see its entries in an agenda below. With All selected, an attended item's dot only shows on a day already past — not on today or a future date, since a done thing's date no longer means anything going forward.
  • Stats (under ⋯): a forward-looking snapshot of the backlog — what's expiring soon, what's past due, what's still someday — plus tap-to-search heatmaps over Category, Tags, and Place. Attended items carry no weight here.
  • Share an item from its detail view — the OS share sheet (WhatsApp, Email, whatever the device offers), or a clipboard copy where that's not available. Text only: name, note, place, links — no attachments.
  • Themes from the sun/moon button — six palettes (Solarized, Monokai Pro…); installs as an app and works offline.
