Click Deck — guide

A standalone web application for tracking graphic narrative and point-and-click adventure games, backed by your own live Notion database.

What it is

Click Deck is a specialized catalog for tracking your adventure-game backlog, what you're currently playing, and what you've finished — from 90s SCUMM classics to today's narrative indies. It leans hard into an industrial, terminal-flavored aesthetic (bracket-notation buttons, scan-line animations, a genuine "NO SIGNAL" static pattern for missing covers) inspired by Beneath a Steel Sky, and it synchronizes live with your own Notion database via this repo's serverless MCP proxy.

Everything is stored in your Notion, under your Notion integration token — Click Deck holds no data of its own beyond a few device-local preferences (theme, weighting mode, banner settings). The app installs as a Progressive Web App (PWA) on desktop or mobile, and once installed its shell — the timeline, analytics and stats views — keeps working offline. Loading fresh data still needs a connection to Notion, of course.

1. Quick Setup

The very first time you open Click Deck with no Notion connection saved, an Onboarding Wizard offers to seed a small local-storage demo collection so you can explore the Timeline/Analytics/Stats views immediately — nothing is written to Notion at this stage, and no Notion token is involved yet (more on that mode in §11).

To connect your own real, persistent collection, set it up directly in Notion, then point the app at it from Settings — there's no in-app "create a database" step:

  1. Create an Integration: Go to Notion Integrations and create a new integration. Call it "Click Deck".
  2. Copy the Token: Copy the Internal Integration Secret — it starts with secret_.
  3. Duplicate the Starter Template: Find the Click Deck — Starter Template page in Notion (ask whoever set up your workspace for the link) and duplicate it into your own workspace — top-right ••• → Duplicate. It comes with the complete schema already in place — see §2 — so there's nothing left to build or patch.
  4. Share the database with your integration: Open the duplicated database, click ••• → Connections, and add your "Click Deck" integration.
  5. Get the Database ID: Copy the database's link. The long hexadecimal string in the URL is its ID.
  6. Connect it: Open Click Deck's Settings (⚙), paste the Integration Token and the Database ID, and save. That's the entire connection step — same on every device, just paste the same two values again.

Using the Watchlist (§5)? There's a second, equally small database and template for that — Click Deck — Followed Studios — Starter Template — duplicated and connected the same way, from its own ID field in Settings.

2. The Notion Database Schema

Whether it's built by the wizard or by hand, Click Deck expects a database with these exact property names. Getting a name slightly wrong (extra space, different casing) means that property silently won't sync.

PropertyTypeNotes
TitleTitleThe game's name. The only property Notion requires by default.
Release YearNumberDrives the Timeline view's sort order and the Stats decade chart.
Developer/StudioSelectOne studio per game. Powers the Analytics STUDIO_MATRIX word cloud.
TagsMulti-selectSee the 5–7 tag policy in §5. Powers the TAG_MATRIX word cloud.
StatusSelectOne of Backlog, Playing, Completed, Abandoned.
RatingNumber1–5. Typically only set once a game reaches Completed.
Journal/NotesRich textYour review. Supports full Notion formatting — see §5.
Current PriceNumber ($)Synced nightly by the pricing cron. Set only for games with a Steam App ID.
Discount PercentNumber (%)Drives the sale banner and the % SALE badge.
Initial PriceNumber ($)The pre-discount price, shown struck-through in the Discounts modal.
Steam App IDNumberPowers cover-art fetching, the Steam store link on every cover, and the pricing cron. See §9.
Price Updated AtDateTimestamp of the last successful pricing sync — surfaced in the Stats screen.
Release StatusSelectComing Soon or Released. Blank/missing is always treated as Released. Drives §5, the Watchlist.
Released AtDateStamped exactly once, the moment a Coming Soon game is observed to have launched. Never set on a game added directly as already-released.
Release DateRich textSteam's own human-readable release date string (e.g. "15 Jul, 2026", or just "Q3 2026" when that's all Steam has). Display only.

All fourteen properties above — base, pricing, and Watchlist — ship together in the Starter Template from §1, so a fresh setup needs no further schema changes. If you're connecting to an older database that predates some of these properties, Click Deck degrades gracefully rather than breaking: it simply omits whatever properties aren't present, and those features activate themselves automatically the moment you add the missing properties directly in Notion (matching the names and types above exactly).

3. Adding Your First Games

Once connected, add games one at a time through the Editor (§6 — click + or press n), or add rows directly in Notion using the schema in §2. There's no bulk-seed action in the app — a curated ten-title starter set (Loom, The Curse of Monkey Island, Sanitarium, Blade Runner, Machinarium, Broken Age, Firewatch, Oxenfree, The Wolf Among Us, Pentiment) exists only inside the local-storage demo mode from §11, to show what a good entry looks like before you've connected anything real — it's never written to Notion.

4. Core Views

Three tabs in the header — [T], [A], [S] — switch between three different lenses on the same collection.

Timeline [T]

The chronologically sorted default view, and the heart of the app. A glowing scan line tracks continuously up and down the vertical track using an IntersectionObserver; individual game cards slide in and their year badges ignite with a neon glow as they enter the viewport, giving the whole page a "living" quality as you scroll. Sort order isn't fixed to release year — a dropdown next to the search box also offers Recently Added, Highest Rated, and Alphabetical; it's fully keyboard-operable (arrow keys move through the options, Enter selects, Escape closes) and, along with the status filter, survives a page reload. A row of status chips ([ALL] [BACKLOG] [PLAYING] [COMPLETED] [ABANDONED], each with a live count) narrows the list further, and stacks with the search box and any active tag filters carried over from Analytics.

Each card shows the cover (click it to open the Steam store page, when a Steam App ID is set), title, developer, status, price, tags, and journal excerpt — plus, for Completed games, five clickable star buttons that update the rating in Notion the moment you click one. A row of quick-status buttons at the bottom of every card lets you move a game between Backlog/Playing/Completed/Abandoned without opening the full Editor.

Analytics [A]

A dashboard for slicing the collection along dimensions the Timeline doesn't surface. Three filter rows — PRICE (Free / <$10 / $10–$30 / $30+), RATING (1★–5★), and ERA (80s/90s/00s/10s/20s) — layer on top of the global search and status filters. A fourth row, TAGS, filters to entries with Under 5 or Over 7 tags — a data-quality tool for spotting entries that drift outside the collection's own 5–7 tag policy (see §5).

Below the filters sit two clickable word clouds — TAG_MATRIX and STUDIO_MATRIX — sized and counted from whatever's currently matching, so clicking a tag drills the whole view (and the Timeline, since the filter is shared) down further. The results render as a responsive gallery grid with hover overlays showing status, year, rating and price. Two export buttons, [EXPORT HTML] and [EXPORT JSON], download whatever's currently filtered: HTML for a clean printable report, JSON as a genuine portable backup of your data (useful precisely because the BYO-token model means there's no other export path).

Stats [S]

A pure data-crunching screen, unaffected by search/filter state — and, like the rest of the app, its numbers never include Coming Soon or Ignored games. Four headline numbers — total entries, average rating, completion rate, and total collection value — sit above six panels: STATUS_MATRIX (a breakdown by play status, plus a COMING SOON line pointing at the fuller detail next to it), WATCHLIST (tracked count, expected-this-year count, recently-released-in-a-year count, ignored count, and a NEXT UP callout naming your single soonest-expected title — with a little animated drum-roll flourish, because anticipation deserves one), FINANCIAL_OVERVIEW (backlog value vs. completed value vs. average price, plus a PRICES LAST SYNCED line so you can tell at a glance whether last night's pricing cron actually ran), TEMPORAL_DISTRIBUTION (oldest/newest release year, and a bar chart of your collection by decade), STUDIO_PROFILES (your most-frequent and highest-rated developers), and TAG_CORRELATIONS (your most-frequent and highest-rated tags — a nice way to notice, e.g., that everything you've rated 5★ happens to be tagged Atmospheric).

5. The Watchlist [W]

A fourth view — the "anteroom" — for anything not yet a fully active part of your collection: unreleased titles you're anticipating, games that just launched, and fresh candidates you haven't decided on yet. The Timeline permanently excludes Coming Soon games (no toggle — [W] is the dedicated place for them), so your main browsing view stays exclusively games you're actually working through.

Groups on one page:

  • Coming Soon — games already added with Release Status: Coming Soon, sorted soonest-expected first, each showing 🔭 EXPECTED: [date] (muted grey for a TBA title, a warning red for one whose expected date has already passed while still unreleased — a subtle colour cue, no extra badges) and a "last checked" age (past 180 days unreleased is flagged as stale, worth a manual look — delayed indefinitely, or quietly cancelled?). Hover a cover for a quick developer/date preview; click it to open the Steam page directly, same as every other cover in the app.
  • Recently Released — anything that flipped from Coming Soon to Released within the last 365 days, most-recent first. These also appear in the Timeline (a NEW ribbon marks them there for their first 30 days), so seeing a game in both places is intentional, not a duplicate. Covers are clickable/hover here too.
  • New Candidates — appears after pressing 🔭 Find New Games (the page's one primary action — it's the button styled to stand out; 🔄 Refresh Release Dates next to it is deliberately understated maintenance), which searches your Followed Studios' Steam catalogs for titles not already in your collection, split into "Not Yet Released" and "Already Released, Not In Your Collection". Results are curated, not chronological: each row shows its matched studio's Personal Value Tier as N filled stars (e.g. ★★★ Wadjet Eye Games for a top-tier studio — no empty stars padding it to a fixed scale), and higher-tier studios always rank above lower-tier ones — only within the same tier does release timing decide order (soonest-expected first for unreleased titles, most-recently-released first for the already-out ones). Every candidate is also checked against your existing collection: an exact Steam App ID match is flagged ✓ ALREADY IN YOUR DB and its Add button is disabled outright, while a title-only overlap (for an existing entry with no App ID on file) is flagged ⚠ POSSIBLE DUPLICATE and just asks you to confirm before adding.
  • Ignored — collapsed by default, showing a count. See below.

A second button, 🔄 Refresh Release Dates, re-checks every Coming Soon game against Steam on demand — the same check the nightly pricing cron also runs automatically, so you never have to wait until 2am UTC to see a launch reflected.

Adding a game only requires a Steam App ID — no rumors or distant-roadmap titles, matching the collection's usual "more inclusive than exclusive" ethos while still requiring something concrete to track. A newly added Coming Soon game intentionally starts with no tags or journal (you can't meaningfully review a game you haven't played); once it flips to Released, a one-time toast nudges you to go fill those in.

Coming Soon and Ignored games are both excluded from the Stats screen and the [R] Random Game roll, and from Analytics regardless of any filter — once released (or un-ignored and released), a game behaves identically to any other entry everywhere in the app.

A separate 🎉 release banner (same shape/animation as the sale banner described below, its own color/icon and its own 24h snooze) appears above the Timeline whenever something's landed recently, and clicking [VIEW] jumps straight to [W].

Ignoring a candidate

Not every candidate Find New Games turns up is one you want to track — a "− IGNORE" button sits next to "+ ADD" on every New Candidates row for exactly that. Ignoring a game isn't just hiding it: it's written to your database (with no tags, journal or price — nothing an ignored game needs) so it genuinely stops being suggested again on future searches, rather than cluttering the same list every time. It also means it's never lost — open the collapsed IGNORED (N) section at the bottom of [W] to see everything you've declined, each with a UN-IGNORE button that re-checks Steam and restores it as Coming Soon or Released (whichever it actually is by now) if you change your mind. An [EXPORT IGNORED JSON] button downloads the full list, so "what did I skip" always has a real answer.

Followed Studios

The list Find New Games searches — configured in Settings → Followed Studios, its own small Notion database (so it syncs across devices, same as your collection). Set up the same way as your main database (§1): duplicate its Starter Template, share it with your integration, paste the ID into Settings. The first time it connects to an empty database, 11 starter studios seed themselves automatically — no button to click. Fully editable afterward from Settings, including an EDIT link on each row for adjusting anything below without leaving the app.

Each studio has four fields:

  • Studio (Title) — the friendly display name, shown everywhere: candidate rows, Settings, exports. It's also the search term "Find New Games" sends to Steam by default.
  • Steam Developer — an optional exact override, only needed when Steam's own listing doesn't match the studio's common name (e.g. "Wadjet Eye" vs. "Wadjet Eye Games", "Microids" vs. "Microïds"). It's never shown outside Settings — its one job is tightening the server-side check that a candidate is genuinely made by that studio before it's ever shown to you. Leave it blank and matching falls back to the Studio name alone.
  • Personal Value Tier — a number ranking how much a new release from this studio deserves your attention (a confidence weighting, not a review score of their whole catalog). No fixed scale is baked into the app — a suggested starting point is 1 (opportunistic follow), 2 (strong follow), 3 (automatic follow), but any numbers work, including a future finer-grained scale. Studios with a higher tier always rank above lower-tier ones in New Candidates, shown as N filled stars (★★★ for a Tier 3 studio, not padded out to some fixed "out of" scale) on every candidate row so you always know why a game surfaced. Leaving it blank ranks that studio last, not zero — it still shows up.
  • Notes — free text, purely for you: why you follow this studio. Shown under its name in Settings.

Worth knowing the difference from a separate, similarly-named field: a game's own Developer/Studio field (in the Editor, §6) is per-game display metadata pulled from Steam at add-time — it isn't formally linked to a Followed Studios row today, so renaming or re-tiering a studio here doesn't retroactively touch games already in your collection.

6. The Game Editor

Click + in the header (or press n anywhere outside a text field) to add a new game; click [E]DIT on any card to edit an existing one. The Editor covers every property in the schema: Title, Cover URL, Release Year, Developer/Studio, Status, Rating (only shown once Status isn't Backlog), Tags, and Journal/Notes.

Cover art: paste an external image URL directly, or click FETCH STEAM to search Steam's store by the current title and auto-fill both the cover URL and the Steam App ID from the best-matching result — not just the literal first hit, since Steam's search is fuzzy-ranked and classics are frequently re-listed with a "Remastered" or "Anniversary Edition" suffix that a plain string compare would miss. Matching the App ID (not just the cover) matters: it's also what makes the cover clickable through to the Steam store page and what the nightly pricing cron uses to find the game at all.

Tags: the picker enforces a maximum of 7 (the button simply does nothing on an 8th click) and shows a live (N/7) count. It also warns — without blocking the save — when an entry has fewer than 5 tags, or when a legacy entry already has more than 7 from before the cap existed, since the collection's own policy calls for 5–7 tags per game: enough for the Analytics word clouds and filters to stay meaningful, not so many that they get noisy. Mix broad genres (Point & Click, Sci-Fi) with specific vibes (Atmospheric, Choices Matter). Any tag already on a game but missing from the canonical picker list (an older or retired tag) still shows up and stays removable — it just won't be offered for new selections.

Journal/Notes & the Expressive Rich-Text Policy

Notes should never read as dry factual summaries ("a game where you click on things") — they're meant to be punchy, opinionated, flavorful mini-reviews that capture what actually made a game land or fall flat. When writing or updating an entry's Journal/Notes (by hand in Notion, or via the dramatize-journal.py script — see §8), lean on real Notion rich-text formatting: bold + color (red, orange, green, purple, pink) for strong emotional or thematic words; italic + gray for sarcasm or subtle mystery; strikethrough or underline for dramatic misdirection. The goal is a reading experience that feels vibrant and dramatically punctuated, not a wall of plain text.

Editing an existing game through the in-app Editor is safe for this formatting as long as you don't touch the Journal/Notes text box — Click Deck detects that the plain text is unchanged and writes the original formatted segments straight back. If you do edit the journal text itself, the Editor shows an inline warning and will save it as plain text on top of whatever was there — full rich-text authoring (colors, bold spans mid-sentence) isn't something a plain textarea can do, so that trade-off is inherent to editing in-app rather than in Notion directly.

7. The Discount Banner & Price Tracking

Whenever any Backlog game has an active discount, a banner appears above the Timeline: 🔥 N GAMES ON SALE! [VIEW]. Clicking it opens a modal with every discounted game, its original and sale price, and a percent-off badge. The banner has a small [X] to dismiss it — doing so snoozes it for 24 hours, after which it reappears automatically if anything's still on sale.

Two Settings controls give you more control over that behavior: a checkbox — "Sale Banner: Always Show" — makes it permanently persistent, ignoring the snooze entirely (the dismiss button hides itself in this mode, since there'd be nothing for it to do). Independent of that toggle, a "Show Sale Banner Now" button clears an active snooze immediately — the safety net for dismissing it by accident and wanting it back right away, rather than waiting out the full 24 hours.

Separately, every time Click Deck loads your collection it quietly compares each game's current price against the last price it saw for that game (kept in a small on-device cache) and toasts PRICE DROP: [title] if anything actually dropped since your last visit — the client-side equivalent of a push notification, since the nightly pricing cron has no way to reach an already-open browser tab.

8. Random Game [R]

Can't decide what to play next? Click [R] for a short slot-machine roll through your Backlog that settles on one game, with [RE-ROLL] to try again and [BEGIN PLAYTHROUGH] to move it straight to Playing (preserving any rating it already had). By default every Backlog game has equal odds. A Settings dropdown — Random Pick Weighting — can instead bias the roll toward your oldest backlog entries (the ones that have been sitting the longest) or your cheapest ones; the active mode shows as a small badge in the roll modal so it's never a mystery why a particular game keeps coming up.

It's off by default (with dozens of cards on screen at once it reads as clutter unless you specifically want the look), and works with any of the six palettes above.

10. Search, Filters & Keyboard Shortcuts

The search box (press / anywhere to jump to it) matches against title, developer, tags, and journal text — so searching "cyberpunk" surfaces both games tagged that way and games whose review happens to mention it, even if the tag itself was omitted. Two more shortcuts: n opens the Editor for a new game, and Escape closes whatever modal or menu is currently open. Both are suppressed while you're typing in any input, textarea or select, so they never fight with normal text entry.

11. Notion Backend, Data Ownership & Sync Errors

Click Deck deliberately doesn't ship with any shared or hardcoded data. It uses a Bring Your Own (BYO) token model: your Notion Integration Token and Database ID live in Settings, are stored only in your browser's local storage, and are sent to Notion's API via a thin, stateless serverless proxy (/api/notion) that exists purely to get around the browser's CORS restrictions — the proxy itself logs and stores nothing. The Followed Studios database from §5 uses the same model, its own database ID stored alongside your main one.

Without a token and database configured, the app falls back to a small local-storage-only demo (the same seed titles from §3), so the Timeline, Analytics and Stats views are fully explorable before you've connected anything real — this demo data is never written to Notion, and switching to a real connection in Settings replaces it entirely.

If a sync ever fails outright — an expired token, a revoked integration, a Notion rate limit — Click Deck shows a distinct SYNC FAILED banner with a [RETRY] button, rather than quietly showing "0 records found" as if your collection were simply empty; those two situations used to look identical, which was more alarming than an actual outage. Day-to-day interactions (rating a game, changing its status, editing or deleting an entry) also update instantly in the interface itself rather than waiting on a full re-fetch of the whole collection — if the underlying write to Notion does fail, the change is rolled back and a toast explains what happened.

12. Steam Pricing & Cover Art

A Vercel-hosted cron job (api/clickdeck-pricing.js) runs nightly against every game that has a Steam App ID, pulling its current price, initial price and discount percentage from Steam's public store API and writing them back to Notion (plus the Price Updated At timestamp). The same nightly call also checks whether any Coming Soon game has launched — see §5. Nothing needs to be running in your browser for this to happen.

Cover art and the Steam App ID itself both come from Steam's search API, matched by title — see §5 for how the in-app "Fetch Steam" button does this, and normalizes away "Remastered"/"Special Edition"/"Anniversary Edition" suffixes so classics still match their modern re-releases correctly rather than matching nothing (or matching the wrong game) on a literal title compare.

13. Administration Scripts

Seven Python utilities in scripts/ talk directly to your Notion database for bulk maintenance that doesn't belong in the UI. Each expects CLICKDECK_NOTION_TOKEN and CLICKDECK_DB_ID as environment variables.

  • backfill-steam-ids.py — finds every game missing a Steam App ID and searches Steam for the best-scoring confident title match (same normalization and 50%-overlap confidence threshold as the in-app fetch), skipping rather than guessing when nothing clears the bar. Useful for entries added directly in Notion without ever going through the Editor.
  • verify-steam-names.py — the read-only counterpart: checks every game that already has a Steam App ID against what Steam actually reports for that ID, flagging title or developer/publisher mismatches (including known studio aliases, e.g. LucasArts/Lucasfilm) without changing anything.
  • backfill-covers.py — finds every game that has a Steam App ID but no Notion page cover set yet (the "NO SIGNAL" placeholder), and patches the cover to that App ID's Steam header image. Since the App ID is already trusted, this is purely mechanical — no title search involved.
  • force-sync-prices.py — an on-demand version of the nightly pricing cron, for when you don't want to wait until 2am UTC to see a price update reflected.
  • dramatize-journal.py — pulls any plain-text Journal/Notes and injects the expressive rich-text formatting described in §6 using a dictionary of thematic keyword-to-color mappings, safely skipping entries that are already manually styled. Highlights at most 3 keywords per entry, so a caption-length note with several theme words doesn't end up a third colored text.
  • sync-watchlist-releases.py — an on-demand version of the release-check half of the nightly cron, for triggering a Coming Soon → Released check without waiting for the scheduled run.
  • validate-watchlist.py — read-only sanity check for the Watchlist feature: flags a Coming Soon game missing its Steam App ID, a status/Released-At contradiction, a missed flip Steam already reports as released, a pull-back (a Released game Steam now shows as unreleased again), and stale entries. Worth running periodically, same spirit as verify-steam-names.py.

14. Disconnecting

There's no dedicated "factory reset" action. Since your actual game data lives in Notion, not in the browser, disconnecting is just clearing the Token and Database ID fields in Settings and saving — that drops you back to the Onboarding Wizard's local-demo mode without touching your collection. To clear the local-demo data itself (if you'd explored that mode first), clear this site's storage from your browser's own settings.

15. PWA & Offline Support

Click Deck ships a web manifest and a scoped service worker, so it's installable as a standalone app on both desktop and mobile, and its shell keeps working without a network connection once installed: the app boots, the timeline/analytics/stats views render, and your last-synced data is still there. What doesn't work offline is fetching fresh data from Notion, or anything that hits the Steam API (cover fetching, pricing). JetBrains Mono is loaded from Google Fonts rather than self-hosted, so the very first load still needs a connection to render with the right typeface; after that it's cached too.
