# Click Deck

**[coneofcold.vercel.app/click-deck-react.html](https://coneofcold.vercel.app/click-deck-react.html)**

Click Deck is a standalone web application designed for cataloguing and tracking graphic narrative and point-and-click adventure games from the 90s to the present. It was originally vibe-coded with Gemini Pro in AntiGravity IDE, then taken over, audited and substantially extended under Claude Code — this file tracks the app as it actually is today, not just its original brief.

For the step-by-step, end-user-facing walkthrough (Notion setup, the full database schema, every view and control explained), see [`public/click-deck-guide.html`](public/click-deck-guide.html) — linked from the app itself and from the Cone of Cold catalog. This file is the technical/architectural companion for anyone (human or Claude) working on the code.

## Architecture

- **Stack**: React, Vite, and standard global CSS (`src/click-deck/styles/theme.css`) for layout, resets and the theme-variable system. Each component keeps its styling in a `<style>` block scoped by its own `cd-*` class prefix rather than CSS Modules.
- **Aesthetics**: A stark, industrial aesthetic initially inspired by *Beneath a Steel Sky*. It utilizes `JetBrains Mono` terminal fonts (loaded from Google Fonts, not self-hosted), sharp edges with zero border radius, bracket-notation UI (`[T]`, `[E]DIT`, `[X]`), and a high-contrast palette. The header logo (`public/click-deck-logo.svg`) is a small transparent SVG built from the same bracket-and-cursor visual language, replacing an earlier opaque PWA-icon PNG that carried a visible background square in the header.
- **Theme System**: Every visual accent in the app is driven by a handful of CSS custom properties (`--cd-accent-cyan`, `--cd-accent-amber`, `--cd-bg-*`, `--cd-status-*`, etc.) redefined per `[data-theme="..."]` value on `<html>`. Six palettes exist today:
  - **Union City** (`union`, default): Cyan/Amber — classic cyberpunk.
  - **Voodoo** (`voodoo`): Toxic Green/Purple — Monkey Island inspired.
  - **Noir** (`noir`): Grayscale/Gold — Grim Fandango inspired.
  - **Sierra Nights** (`sierra`): Indigo/Gold — King's Quest-era Sierra VGA.
  - **Amber Terminal** (`amber-terminal`): Amber/Burnt Orange — classic monochrome DOS/BBS CRT, echoing the app's own terminal chrome.
  - **CGA** (`cga`): Cyan/Magenta — early-90s DOS composite CGA.
  - A separate, independent Settings toggle — **Retro CRT Mode** (`data-crt="true"` on `<html>`, `cd_crt_effect` in localStorage) — layers scanlines, a screen vignette, and viewfinder-style corner brackets (`.cd-panel::before/::after`) on top of whichever theme is active. Off by default: with dozens of `.cd-panel` cards on screen at once (every game card *is* one), it reads as clutter unless deliberately opted into.
  - Play-status colors (`--cd-status-backlog/playing/completed/abandoned`) are defined once in `:root` in terms of the theme accents, so they automatically re-derive per theme via normal CSS cascade/`var()` resolution — no per-theme duplication needed.
  - A single global `@media (prefers-reduced-motion: reduce)` rule at the end of `theme.css` collapses every animation/transition app-wide (banner pulses, icon spin, drum roll, CRT flicker/static, the `[R]` roll) to effectively instant, rather than gating each keyframe individually.
- **Backend**: Live Notion Database Integration (`3a3d3e6d-60db-81b2-a8d9-ca78e8100ef8`) via the repo's stateless `/api/notion` proxy (`api/notion.js`) — see [Notion Backend](#notion-backend--database-initialization) below.
- **Automated Pricing & Discounts**: A Vercel-hosted cron job (`api/clickdeck-pricing.js`, `0 2 * * *` in `vercel.json`) nightly syncs real-time Steam Store prices, initial prices and discount percentages for every game with a `Steam App ID`. `Price Updated At` is stamped on *every* game it successfully checks that night — it's the "did we check" signal, not a "did the price move" one — while `Current Price`/`Initial Price`/`Discount Percent` only get rewritten when they actually differ (a small floating-point tolerance guards against noise), so a stable-priced game doesn't churn Notion's edit history every night but still shows an up-to-date sync date. A freshly-added game (or one just flipped Released via the Watchlist) simply waits for its first cron cycle, same as everything else — there's no separate "new row" fast path.
  - The same nightly run also does two more things unrelated to Steam pricing: the Watchlist flip-time enrichment (tags/journal/developer/cover derivation — see [Watchlist](#watchlist-w-anteroom) below), and a third, fully independent pass that **auto-dramatizes journal entries** — a JS port of `scripts/dramatize-journal.py` (`api/_lib/clickdeckDramatize.js`, kept in sync with the Python script the same deliberate-duplication way `steamMatch.js`/`backfill-steam-ids.py` are) runs against *every* game with a non-empty `Journal/Notes` (not just App-ID-having ones — this pass needs no Steam call at all), skipping any entry that already carries one of the theme's 7 colors and highlighting up to 3 keyword matches per entry otherwise. It walks the *existing* rich_text runs rather than flattening and rebuilding from plain text, so prior manual formatting (e.g. bold added by an editorial pass) survives untouched on top of the new color. This closes the gap where a Steam-derived or hand-written journal entry (from a Watchlist flip, a direct add, or an Editor edit) used to sit unstyled forever unless someone remembered to run the script by hand.
- **Client-side data layer**: `src/click-deck/lib/mcp-connector.js` is the single point of contact between the UI and storage — it transparently switches between the real Notion backend (when a token + database ID are configured) and a `localStorage`-only demo mode (see [Notion Backend](#notion-backend--database-initialization)). A Notion sync failure (bad token, revoked integration, rate limit) is no longer swallowed to an empty array here — it propagates to the caller, so `App.jsx` can show a distinct `SYNC FAILED` state instead of it looking identical to a genuinely empty collection.
- **Optimistic UI**: Rating changes, status changes, edits and deletes update React state immediately rather than re-fetching the entire (paginated) collection from Notion after every single mutation. If the underlying write actually fails, the optimistic change is rolled back and a toast explains what happened. This lives in `App.jsx`'s `handleSaveGame`/`handleUpdateStatus`/`handleDeleteGame`.
- **Test Suite**: A comprehensive Vitest + React Testing Library component suite (260+ tests across `src/click-deck/`) covering every component and library module, including dedicated coverage for the rich-text preservation logic, the Steam/HLTB title-matching heuristics, optimistic-update rollback, and every Settings control.

## Scripts & Automation

Located in the `scripts/` directory, these Python utilities interact directly with the Notion backend to automate and verify collection data. Each expects `CLICKDECK_NOTION_TOKEN` and `CLICKDECK_DB_ID` as environment variables (or hardcoded at the top of the script for a one-off run).

- **`backfill-steam-ids.py`**: Finds every game missing a `Steam App ID` and searches Steam's storesearch API for it, writing back the best-matching App ID. Scores *every* candidate by normalized-title overlap (see `containment_score`/`normalize_title`) and takes the best one, rather than the first result that loosely "contains" the query — a short or generic title (`Norco`, `The Dig`, `Loom`) can otherwise coincidentally match an unrelated, much longer Steam listing purely because of result order. A match below a 50% length-overlap ratio isn't treated as confident; the script skips it (printing a note) rather than guessing. This mirrors `src/click-deck/lib/steamMatch.js`, which the in-app Editor's "Fetch Steam" button uses for the exact same reason — the client-side version additionally surfaces a "⚠ Uncertain Steam match" toast when it falls back to an unconfident guess, so a wrong assignment doesn't slip in silently.
- **`verify-steam-names.py`**: The read-only counterpart — verifies games that *already have* a `Steam App ID` against what Steam actually reports for that ID. Cross-references the Notion `Developer` field against both Steam Developers and Publishers (including studio aliases like LucasArts/Lucasfilm) and normalizes titles to prevent false-positive mismatches (same overlap-ratio logic as above), without writing anything back. Worth running periodically as a spot-check even though App ID assignment is more conservative now.
- **`backfill-covers.py`**: Finds every game that has a `Steam App ID` but no Notion page cover set (the "NO SIGNAL" placeholder in Timeline/Analytics), and patches the cover to that App ID's Steam header image. The app's `coverUrl` comes from the Notion *page's* cover image (`page.cover`), which is a separate thing entirely from the `Steam App ID` property — a game can have a fully correct App ID and still show no cover if it was added directly in Notion, or added before the app had a "Fetch Steam" button. Since the App ID is already trusted for these rows, this is purely mechanical — no fuzzy title search, no ambiguity.
- **`force-sync-prices.py`**: An on-demand version of the nightly `api/clickdeck-pricing.js` cron, for triggering a price refresh without waiting for the scheduled run. Shares its "resolve what changed" logic with the cron via `api/_lib/clickdeckPricing.js`, which has direct unit test coverage (`api/_lib/clickdeckPricing.test.js`).
- **`dramatize-journal.py`**: The manual/ad-hoc counterpart to the nightly cron's own dramatization pass (see the Automated Pricing bullet above) — same keyword-highlight logic (a predefined dictionary of thematic keywords, e.g. coloring "horror" red, "cyberpunk" purple), same `MAX_HIGHLIGHTS_PER_ENTRY = 3` cap (validated against the live collection's actual journal text — several entries hit 5–6 matches in a single short paragraph without a cap, coloring close to a third of the text and undercutting the "punchy accent" goal rather than serving it), same detection rule (skips an entry only if it already carries one of the theme's actual colors — plain manual bold with no color doesn't count as "already styled" and is still processed, preserving that bold while adding the highlight). Since the cron now runs this automatically every night, this script exists purely for triggering it on demand without waiting for the scheduled run — same relationship `force-sync-prices.py` has to the cron's price-checking half. Configured strictly via environment variables (`CLICKDECK_NOTION_TOKEN` / `CLICKDECK_DB_ID`, with a hardcoded local fallback for one-off runs) and excluded from version control via `.gitignore`.
- **`sync-watchlist-releases.py`**: An on-demand version of the release-check half of the nightly pricing cron (see [Watchlist](#watchlist-w-anteroom) below) — for triggering a Coming Soon → Released check without waiting for the scheduled run. Mirrors `force-sync-prices.py`'s relationship to the cron's price-checking half; shares its flip rules with `api/_lib/clickdeckWatchlist.js`, kept in sync deliberately rather than imported (same convention as `backfill-steam-ids.py` mirroring `steamMatch.js`).
- **`validate-watchlist.py`**: The read-only sanity check for the Watchlist feature — the primary safety net, meant to run after the first production cron run and periodically after. Flags: a Coming Soon game missing its Steam App ID (violates the "must already have an App ID" rule); a `Released At` set while still marked Coming Soon (contradiction); a Coming Soon game Steam already reports as launched (a missed flip); a `Released` game Steam now reports as `coming_soon: true` again (a pull-back — never auto-reverted, only ever flagged here for manual review); a `Released` row stamped very recently whose Steam date looks much older (would indicate a direct-add incorrectly backfilling `Released At` instead of leaving it null); and stale Coming Soon entries whose Release Year is well in the past. Writes nothing back to Notion.
- **`backfill-hltb.py`**: Finds every game missing a `Length (hrs)` and searches HowLongToBeat for it, writing back the "Main + Sides" completion time (HLTB's own internal field is `comp_plus`/"Main + Extra" — same number, friendlier label). Mirrors `api/clickdeck-hltb.js`'s reverse-engineered two-step handshake exactly (see that file's header comment) and `src/click-deck/lib/hltbMatch.js`'s title-confidence scoring (same convention as `backfill-steam-ids.py` mirroring `steamMatch.js`). Skips games that already have a length — this is a one-off fill-the-blanks pass; use the Editor's **FETCH HLTB** button to fix a bad match on a single game.

Two other root-level scripts (`find-covers.cjs`, `validate-covers.cjs`) are one-off scratch utilities from early development (scraping Wikipedia infobox images as a cover-art workaround, and checking cover URLs for 404s) — not part of any regular workflow, kept for reference only.

## Core Views

Click Deck features four primary perspectives over your game collection, toggled via the main navigation (`[T]` / `[A]` / `[W]` / `[S]`):

### 1. Timeline View (Micro-Animations)
The chronologically sorted timeline is the default and core view of the application. It leans heavily into a retro-futuristic aesthetic by utilizing an `IntersectionObserver` to trigger scroll-based micro-animations.
- A persistent, glowing scan line (colored per the active theme) tracks up and down the vertical timeline track.
- As individual game nodes enter the viewport, they smoothly slide in from the side.
- Year markers and game cards ignite with a neon glow effect, creating a highly dynamic, "living" interface that reacts continuously to user scroll position.
- Sort order is user-selectable (Timeline / Recently Added / Highest Rated / Alphabetical), and stacks with the global search box, a row of status chips (`[ALL]`/`[BACKLOG]`/`[PLAYING]`/`[COMPLETED]`/`[ABANDONED]`, each with a live count), and any tag filters carried over from Analytics. Both the sort mode and status filter persist across reloads (`localStorage` keys `cd_sort_by`/`cd_status_filter`), same as theme and CRT mode. The sort dropdown is fully keyboard-operable: `↓`/`↑` on the trigger opens it focused on the current selection, `↓`/`↑`/`Home`/`End` move (and wrap) through the options, `Enter`/`Space` selects, and `Escape` closes and returns focus to the trigger.
- Global search matches title, developer, tags *and* journal text — not just title/developer as in the original implementation.
- Each card's cover, when the game has a `Steam App ID`, is a real `<a href target="_blank">` to the Steam store page — not a `div` + `onClick={() => window.open(...)}`, which is unreliable across browsers/popup blockers on a real domain (and once the app is installed as a PWA) even though it worked fine on `localhost`. The same fix applies to the Analytics gallery, the Discounts modal, and the Random Game modal's cover.
- A dismissible discount banner (see [Discount Banner & Price Tracking](#discount-banner--price-tracking) below) appears above the timeline whenever any Backlog game has an active sale.

### 2. Analytics View
A high-level dashboard providing advanced multi-dimensional filtering over your collection.
- **Advanced Querying**: Clickable toggle rows for `PRICE`, `RATING`, `ERA`, `TAGS`, `RELEASE`, `COMPLETED`, and `LENGTH` that layer on top of the global `Status` and `Search` parameters. The `TAGS` row (`Under 5` / `Over 7`) is a data-quality filter for spotting entries that fall outside the collection's own 5–7 tag policy — see [Adding a New Game Requirements](#adding-a-new-game-requirements) below. `COMPLETED` (`Any`/`1mo`/`3mo`/`6mo`/`12mo`) filters by how recently a game was marked Completed, via `src/click-deck/lib/completionTracking.js`'s shared window logic (also used by the Stats panel below, so the two can't disagree). `LENGTH` (`Short <4h` / `Medium 4-12h` / `Long 12-25h` / `Epic 25h+`) filters by HLTB "Main + Sides" length, via `src/click-deck/lib/lengthBuckets.js` — see [Completion & Length Tracking](#completion--length-tracking) below for where both fields come from.
- Features a **Dynamic Tag Word Cloud** (`TAG_MATRIX`) and a **Developer Word Cloud** (`STUDIO_MATRIX`) that let you visually drill down by genre or studio; clicking a chip filters the whole app (the filter is shared with the Timeline).
- Displays a responsive Gallery Grid featuring game metadata (Release Year, Price, Rating, and Status) that dynamically updates based on the active query. Each grid item's cover is a real Steam link, same as the Timeline.
- Supports exporting the currently filtered dataset as a clean HTML report (`[EXPORT HTML]`) or as portable JSON (`[EXPORT JSON]`) — the JSON export exists specifically because the BYO-Notion-token model means there's no other built-in backup path.

### 3. Stats Screen
A pure data-crunching dashboard designed specifically for adventure game collectors, denoted by the `[S]` icon. Unaffected by the global search/status filters (it always summarizes the whole collection).
- Calculates your exact collection completion rate, mean average rating, and **Total Collection Value** in glowing amber.
- Plots your games on a timeline bar chart by decade.
- Profiles your top and highest-rated Developer Studios.
- Analyzes which tags and genres consistently earn the highest ratings.
- A **Financial Overview** breaks down the monetary value tied up in Backlog vs. Completed games, and now shows **`PRICES LAST SYNCED: [date]`** — the most recent `Price Updated At` across the whole collection — computed client-side from data that was previously captured from Notion but never surfaced anywhere, leaving no way to tell whether the nightly pricing cron was actually still running.
- **`COMPLETION_VELOCITY`**: how many games you've marked Completed in the last 30 days / 3 / 6 / 12 months, plus an all-time count — pacing at a glance, not just a lifetime total. A Completed game with no `Completed At` date (a pre-R2 entry, or one marked Completed before this schema patch) still counts toward all-time but can't match a window; the panel surfaces that gap with an asterisk rather than silently under-counting.
- **`PLAYTIME_ANALYSIS`**: total Backlog hours vs. Completed hours (an honest "hours ahead of you" / "hours behind you" pair, not one meaningless sum), average/longest/shortest length, and a bucket bar chart across the same Short/Medium/Long/Epic buckets Analytics' `LENGTH` filter uses — all from HLTB "Main + Sides" data, see below.

## Watchlist ([W] Anteroom)

A fourth view for anything not yet a fully active part of the collection — unreleased titles being tracked pre-release, games that recently launched, and fresh discovery candidates not yet decided on. Fully designed and hardened before any code was written (see project memory `click-deck-watchlist-plan` for the full design history); summarized here as-built.

- **Schema**: three new properties on the main DB — `Release Status` (select: `Coming Soon`/`Released`/`Ignored`, blank always reads as `Released` — no backfill needed on pre-existing rows), `Released At` (date, **stamped only on an observed Coming Soon → Released transition, never on a direct add of an already-released game** — this is the load-bearing rule that keeps the Recently Released banner/section honest), and `Release Date` (rich text, Steam's own display string, refreshed on every check). No in-app patch button — these ship baked into the [Click Deck — Starter Template](#notion-backend--database-initialization) database, the single source of truth for schema going forward. `mcp-connector.js`'s `mapPageToGame`/`mapGameToProperties` key these three fields off actual Notion property *presence* (not just value) — a game loaded against an older, unpatched database gets them as `undefined`, which `mapGameToProperties` correspondingly omits from every write, so an ordinary edit never references a Notion property that doesn't exist and can't break saves. `readReleaseStatus()`/`isComingSoon()` (the read side of that same default-to-Released rule) live in their own module, `src/click-deck/lib/releaseStatus.js`, deliberately *not* part of `mcp-connector.js` — `App.test.jsx` mocks that whole module, and any export added there would silently resolve to `undefined` under that mock.
- **Followed Studios**: a second, small Notion database (`Title` + optional `Steam Developer` exact-tag override + `Notes` + `Personal Value Tier`), its own id in `localStorage` (`cd_studios_db_id`), managed via `src/click-deck/lib/studios-connector.js` — same BYO-token/local-demo-fallback shape as `mcp-connector.js`. Its Settings panel is a plain "paste an existing database ID" field (mirroring the main Database ID field) rather than an in-app "create a new one" button, so switching devices means pasting the same id rather than accumulating a fresh database per device. Fully editable from Settings, including an inline **EDIT** on each row for adjusting tier/notes/Steam Developer as opinions change; the first successful connection to an empty database silently seeds 11 starter studios (no button — a freshly duplicated Starter Template always starts empty, so this is just what "first connection" looks like). Named "Click Deck — Followed Studios" to pair with the main "Click Deck" DB under the same heading in the Notion workspace's App Databases section; it has its own [Starter Template](#notion-backend--database-initialization) too.
  - **`Title` vs. `Steam Developer`** answer two different questions and are used for two different purposes. `Title` is the studio's *display* name — what shows up everywhere in the app (candidate rows, Settings, exports) — and is also the default search term "Find New Games" uses against Steam's store search. `Steam Developer` exists only because Steam's own `developers`/`publishers` fields on a game don't always match a studio's friendly name (the live collection already has both "Wadjet Eye" and "Wadjet Eye Games", "Microids" and "Microïds") — it's an optional exact-tag override used purely by `studioMatchesCandidate()`'s server-side verification step, never shown to you beyond Settings. Fill it in only when a studio's Steam tag genuinely differs from its `Title`; leaving it blank falls back to matching on `Title` alone. Separately, a *game's* own `Developer/Studio` field (on the main Games DB) is a per-game display value populated from Steam's `developers[0]` at add-time — usually similar to a followed studio's `Title` but not formally linked to it today.
  - **`Personal Value Tier`** is a plain number (no fixed enum — the app never assumes exactly three levels), read as a confidence weighting rather than a quality score: how much a *new* release from this studio deserves your attention, not how good you think their catalog is overall. Gabriel's current scale is 1 (opportunistic follow) – 3 (automatic follow), but the field tolerates any number, including a future 5-level or fractional scale, without any code change — `normalizeTier()` in `studios-connector.js` is the one place that reads it, defaulting an unset tier to the lowest weight so untiered legacy studios still surface, just last. This is what turns Discovery from chronological into curated: see below. Deliberately shown as a *colour accent*, never stars — stars already mean "rating" elsewhere in the app (the Editor's 1-5★ field), and this measures a different thing entirely (`tierAccentColor()` in `studios-connector.js`, bucketed muted/cyan/amber by weight, shared by both the Settings studio list and New Candidates rows). The colour alone collapsed in the grayscale Noir theme (and even outside Noir, Tier 1 and "no tier" rendered as the literal same muted colour) — `tierBorderStyle()` adds a theme-independent width/style on top: T3 solid-thick, T2 solid-thin, T1 dashed, unset a plain hairline.
- **Release-status checking**: one shared resolver (`api/_lib/clickdeckWatchlist.js`, mirrored client-side in `src/click-deck/lib/watchlistResolver.js` for the manual action — same deliberate-duplication convention as `backfill-steam-ids.py`/`steamMatch.js`), run from two triggers — the nightly pricing cron (`api/clickdeck-pricing.js`, its Steam `appdetails` call widened from `price_overview`-only to `basic,price_overview,release_date,genres` so release info, cover art and genre data all come back in the same request that already fetches price) and a manual **Refresh** icon on `[W]` (via `api/steam-search.js`'s `?appids=` mode, widened the same way — merged in from a dedicated `clickdeck-appdetails.js` proxy to stay under Vercel Hobby's 12-function cap, see below — plus `src/click-deck/lib/watchlistActions.js`'s `refreshComingSoonGames()`, which applies flips through the normal `McpConnector.updateGame` path). Flip rule: only a game currently `Coming Soon` whose Steam `release_date.coming_soon` reads exactly `false` flips — price presence is never used as a proxy (a priced pre-order still reports `coming_soon: true`). A pull-back (a flipped game later reporting `coming_soon: true` again) is never auto-reverted, only flagged by `validate-watchlist.py`.
  - **Flip-time enrichment**: a Coming Soon add always lands with empty `tags`/`journal` (see Discovery below) — the moment the resolver observes the actual flip to `Released`, it also derives `Tags`, `Journal/Notes` (from Steam's `short_description`), and `Developer/Studio` (from Steam's `developers[0]`), **but only when the field is still blank** — a game hand-tagged, hand-noted, or with its developer already set (the normal case — add-time already sets it) keeps exactly what's there, never overwritten.
  - **Tag inference, specifically** (`api/_lib/clickdeckTagInference.js`, client twin `src/click-deck/lib/tagInference.js`): matches Steam's genres + description text against Click Deck's own curated ~118-tag vocabulary (`ALL_TAGS` in `seed-data.js`), not a raw genre-list passthrough — Steam's genres alone are broad and generic (Adventure, Indie, RPG, Casual are the only ones with a direct ALL_TAGS match; everything else Steam might return — Action, Simulation, Strategy, ...— is silently dropped rather than invented). The rest of the signal comes from scanning `short_description`/`about_the_game`/`detailed_description` (HTML stripped first) for the same kind of specific, flavorful language a human curator would use — noir, psychological horror, choices matter, southern gothic, and ~90 more phrase-to-tag mappings, each reviewed for common false-positive wording (a bare "cult" would wrongly fire on "cult classic"; the trigger requires the plural "cults"/"cultists"; "android" was dropped as a trigger for Automata since it collides with "available on Android" the OS). `Mature` is inferred separately, from Steam's `required_age` field rather than guessed from marketing copy; `Female Protagonist` and `LGBTQ+` are deliberately excluded from inference entirely — identity attributes aren't something to guess at from promotional text, these stay human-assigned only. Verified live against real Coming Soon titles: 4–7 accurate tags per game vs. 2 with the old raw-genre approach; a thin Steam page can still land under 5 (honestly — no invented tags to force the floor), which is exactly what Analytics' existing `TAGS: Under 5` filter is for. This same inference function also replaced the old raw-genre passthrough in Discovery's already-released add path (`api/clickdeck-studio-search.js`). None of this was possible via the API until a one-time schema sync (2026-07-23): `ALL_TAGS` had 20 entries (Folklore, Melancholy, Whodunit, Heist, Romance, Philosophy, Wholesome, Espionage, Cults, Amnesia, Artificial Intelligence, Visual Novel, Interactive Fiction, Escape Room, Voxel, Low Poly, Cel-Shaded, Time Loop, Non-Linear, Branching) that were never actually registered as real Notion `multi_select` options — writing any of them 400'd the *entire* PATCH (Notion write validation is atomic per request), silently failing release-status/cover/price updates on the same page too. Fixed via one `ALTER COLUMN "Tags" SET MULTI_SELECT(...)` DDL statement on both the live Games DB and the Starter Template, listing all 118 tags with the 98 existing ones' exact original colors preserved (verified after the fact that no existing game's tags were disturbed). Two fields are deliberately left OUT of this derivation, on purpose, not by oversight: **price** — the same Steam response carries `price_overview`, but the pricing cron recomputes price/discount separately in the very same run, so re-deriving it in the flip resolver would just be redundant (the manual "Refresh Release Dates" path has no pricing pass of its own and is fine leaving it to that night's cron); **Length (hrs)** — HowLongToBeat almost never has real submission data for a game on its actual release day, and its reverse-engineered scrape (`api/clickdeck-hltb.js`) is already the single most fragile integration in the app, so it deliberately stays a manual/scripted backfill (Editor's FETCH HLTB button, `backfill-hltb.py`) run once real data has had time to accumulate, rather than one more moving part wired into the nightly cron for near-zero day-one payoff. Cover art gets the more generous treatment: `coverUrl` is refreshed from Steam's `header_image` on **every** check while a game is still `Coming Soon`, not just at the flip — early listings often carry a generic placeholder that Steam swaps for real key art at some point during development, and this refresh (both the cron and the manual path) is what actually surfaces that once it exists. Cover art is a Notion *page-level* field, not a regular property, so both write paths PATCH it separately from the properties write (`cover` alongside `properties` in the cron's single PATCH; a dedicated `McpConnector.updateGameCover` call client-side) — this also closed a pre-existing latent bug where `unignoreGame`'s own cover refresh silently never reached Notion for the same reason.
- **Header actions** (both optional now that the cron runs daily): the `[W]` header is a single row — the stat line (`N tracked · N expected this year · N just released`) leads, and two small, muted, label-less SVG icon buttons sit quietly at the far end: a magnifier (**Find New Games**, discovery) and circular arrows (**Refresh Release Dates**, a manual re-check). Both were prominent labelled buttons during development but are now deliberately understated — the daily cron does the real work, so these are optional nudges, not primary actions. The icon is the whole affordance (no hint text); the accessible name lives in `aria-label`/`title`, and the icon spins while its action is in flight (`is-busy`).
- **Discovery**: the **Find New Games** icon on `[W]` calls `api/clickdeck-studio-search.js`, which searches Steam per followed studio and *verifies* each candidate's actual `developers`/`publishers` fields server-side (via a locally-duplicated, diacritic-stripping normalizer — `Microïds` and `Microids` need to match) rather than trusting the search term alone; candidates are split into Not Yet Released vs. Already Released, Not In Your Collection. The client then runs its own two-tier dedupe (`steamMatch.js`'s `findDuplicateInCollection`) against the *entire* collection: an exact Steam App ID match is a hard block (Add disabled outright), while a normalized-title match against a collection row with no App ID on file (a real gap — some legacy entries predate "Fetch Steam") is a soft block requiring an explicit confirm. Only candidates with a confirmed Steam App ID are ever addable — no roadmap rumors. `src/click-deck/lib/watchlistActions.js`'s `candidateToNewGame()` builds the add payload; a Not-Yet-Released pick lands as `Coming Soon` (no `Released At`), an Already-Released one lands as `Released` directly, skipping Coming Soon entirely.
  - **Curated, not chronological**: each candidate carries its matched studio's `Personal Value Tier` straight through from the server (`studioTier` on the candidate), and both New Candidates lists are sorted **tier first** (a Tier 3 studio's game always outranks a Tier 1 one, regardless of release timing), **release date second** (soonest-expected first within Not Yet Released, most-recently-released first within Already Released) — `compareCandidates()` in `watchlistActions.js`. Every candidate row carries a coloured, theme-independently-styled left-border accent (`tierBorderStyle()`) reflecting its studio's tier, with the exact number available on hover (`title` attribute) — a deliberately quiet signal rather than a number or star count competing for attention with the rest of the row.
- **`[W]` view** (`src/click-deck/components/WatchlistView.jsx`): **Coming Soon** (sorted soonest-expected first: parsed year, then Steam's date string, then add-date; a `last checked` age derived from `Price Updated At` flags anything unreleased for 180+ days as stale, worth a manual look; `formatExpected()` decides the `EXPECTED:` line's text and colour together — a Steam `Coming soon` string is folded to just **Soon** and coloured like `TBA` (both are vague-bucket, and repeating "Coming soon" verbatim under the COMING SOON header is noise); otherwise muted for `TBA`, the theme's danger red for **overdue** (an expected date/year already passed while still Coming Soon), the normal accent otherwise — backed by a "⚠ (OVERDUE)" text marker, not colour alone, so the signal survives grayscale/colourblind viewing too); **Recently Released** (`Released At` within 365 days, most-recent first — the window is intentionally a full year, not 30 days, so the first production run already has real material to check); **New Candidates** (only after a search, ephemeral, never persisted until explicitly added); and a collapsed **Ignored** section (see below). Covers in both Coming Soon and Recently Released are real links straight to the Steam store page, same as everywhere else in the app, with a hover overlay (`WatchlistCover`, shared by both grids) surfacing the developer + expected/actual release date.
- **Ignoring a candidate**: a **"− IGNORE"** button sits next to **"+ ADD"** on every New Candidates row (styled with a deliberately neutral/muted palette color, not the amber "exact duplicate" or red "possible duplicate" already in use, shifting to `--cd-status-abandoned` only on hover/focus to read as a dismissive action). Ignoring still writes a real DB row — `Release Status: Ignored`, no tags/journal/price — via `candidateToIgnoredGame()`, so an ignored title genuinely stops being suggested again (it's caught by the same appId-exclude + fuzzy-title dedupe as anything else already in the collection) rather than needing to be remembered or re-filtered client-side each search. Ignored games are hard-excluded from Timeline, Analytics (regardless of its `RELEASE:` filter), Stats and Random — enforced once, upstream, via `App.jsx`'s `baseFilteredGames` (which now filters `!isIgnored(g)` before anything else runs) and `isActiveCollectionGame()` (excludes both `Coming Soon` and `Ignored` in one predicate, used by Stats/Random so neither view can drift from Timeline on what counts as "active"). They remain visible and exportable from `[W]`'s own collapsed **IGNORED (N)** section — a **"UN-IGNORE"** button per row re-checks Steam (`unignoreGame()` in `watchlistActions.js`) and restores the game to `Coming Soon` or `Released` depending on what Steam now reports, refreshing price/cover/date in the same pass; a direct restore to `Released` never stamps `Released At`, matching the rule that only an *observed* Coming-Soon transition does. When it resolves to `Released`, it also derives tags/journal/developer from Steam the same way the flip resolver does (see Flip-time enrichment above) — an Ignored game was always saved with empty tags/journal, so restoring it deserves the same starting point a flip gives, never overwriting anything already filled in. An **"[EXPORT IGNORED JSON]"** button downloads the full ignored list (title, developer, App ID, release date, and `createdTime` as an "ignored around" timestamp — no separate property needed for that, since these rows are only ever created fresh at ignore-time).
- **Timeline**: hard rule, no toggle — permanently excludes `Coming Soon` (`App.jsx`'s `timelineGames`/`timelineEligibleGames`, derived separately from what's passed to Analytics so Analytics can still optionally include Coming Soon; `Ignored` is excluded even earlier, upstream of both). A game that flips to `Released` appears in Timeline automatically; for its first 30 days there it also carries a `NEW` ribbon (`isRecentlyReleasedWithinDays`, a tighter window than the banner's 365 days) so the Timeline/`[W]` overlap during that window reads as "just landed," not duplicated.
- **Analytics**: a `RELEASE:` filter row (`All`/`Released`/`Coming Soon`, defaulting to `Released`) alongside the existing `PRICE:`/`RATING:`/`ERA:`/`TAGS:` rows — `Ignored` was deliberately left off this filter's option list, since those games never reach Analytics' props at all regardless of filter choice. The `TAGS:` data-quality filter (`Under 5`/`Over 7`) unconditionally excludes `Coming Soon` games regardless of the RELEASE filter — the 5–7 tag policy is for played/owned games, and an unplayed watchlist entry ships with no tags by design, so without this exclusion it would be a permanent, unfixable false positive.
- **Stats**: `StatsView` now takes two separate arrays — `games` (already Coming-Soon/Ignored-filtered by `App.jsx`, everything the existing collection stats are computed from, unchanged) and a new `watchlistGames` prop (the full unfiltered array, used only to compute the Watchlist numbers below, so neither leaks into the other). `STATUS_MATRIX` gets one added line, `COMING SOON`, as an at-a-glance pointer to the fuller detail in a new, separate **WATCHLIST** panel: tracked count, expected-this-year count, recently-released-in-365-days count, ignored count, and a **NEXT UP** callout naming the single soonest-expected Coming Soon title (via the same `sortComingSoonSoonestFirst()` `[W]` itself uses, moved into `releaseTracking.js` so the two can't disagree on which game is "next") — decorated with a small looping three-dot `DrumRollIcon` SVG (exported from `WatchlistView.jsx`) for a touch of anticipation.
- **Random Game**: excludes both `Coming Soon` and `Ignored` via the same `isActiveCollectionGame()` predicate as Stats.
- **Editor**: a `Release Status` dropdown, shown only once the schema is known to be patched (`watchlistSchemaReady` in `App.jsx`, computed as "does any loaded game carry the property" — a database-level property, so one hit means all rows have it) or the game being edited individually already carries the field — the same undefined-omission safety as the mapper. The tag-count warning is suppressed while a game is `Coming Soon`, for the same reason as the Analytics data-quality filter above.
- **Release banner**: `src/click-deck/lib/releaseTracking.js` holds the shared predicate (`isRecentlyReleased`, `Released At` non-null within 365 days) used by both the banner and `[W]`'s Recently Released group, so the two can never disagree about which games count. Same visual structure/animation as the sale banner (inlined in `App.jsx`, not a separate component, matching how the sale banner itself is built) but its own accent color, its own icon, and an independent 24h snooze key (`cd_release_snooze_until`) plus a Settings "Show Release Banner Now" recovery button and a persistent-mode checkbox — full parity with the sale banner's controls. Clicking `[VIEW]` navigates to `[W]` rather than opening a separate modal. A one-time "🎉 X just released — add tags & notes?" toast fires the first time each game is seen as recently released (`takeUnnudgedRecentlyReleased`, a `localStorage` set of already-nudged ids) — now a prompt to *refine* the flip's auto-derived Steam tags/description into your own take, rather than fill a genuinely empty gap (the resolver itself fills that at flip time, see above).
- **Validation**: `scripts/validate-watchlist.py` (see Scripts & Automation above) is the primary safety net, meant to be run after the first production check and periodically after.
- **Known-fixed bug**: "last checked" on Coming Soon cards used to be permanently stuck on "never" — root cause was that Steam's `appdetails` endpoint silently dropped support for its comma-separated multi-appid form (confirmed live, HTTP 400 either way), which both the nightly cron and `force-sync-prices.py` still used, so `Price Updated At` was never actually being written. Both now issue one request per App ID.
- **Known-fixed bug (2026-07-23)**: the nightly cron's own Notion query for "every game with a Steam App ID" (`api/clickdeck-pricing.js` step 1) had no pagination — a single `page_size: 100` request with no `has_more`/`next_cursor` loop, so once the collection grew past 100 App-ID rows, everything past the 100th was silently never checked at all: no price sync, no release-flip, none of the flip-time enrichment above. Confirmed live: 121 games had a Steam App ID, only the first 100 were ever processed. The sibling Python scripts (`force-sync-prices.py`, `sync-watchlist-releases.py`) already paginated correctly — this was isolated to the deployed JS cron. Fixed by paginating the same way; verified end-to-end by invoking the real handler locally against production (`processed` went from 100 to 121).
- **Known-fixed bug (2026-07-23)**: a Coming Soon game with no pre-order price announced on Steam yet showed `$0.00` in `Current Price` instead of staying blank — `resolvePriceUpdate` (`api/_lib/clickdeckPricing.js`) treated "Steam returned `success: true` with no `price_overview`" as always meaning "free", writing a literal `$0.00`. That condition also covers "no price to report yet" and "delisted", which aren't the same thing as confirmed-free. Fixed using Steam's own `is_free` flag to tell the two apart: genuinely free still writes `$0.00`; anything else with no `price_overview` now leaves price fields untouched (still stamps `Price Updated At` — that field means "did we check", not "did we find a price"). The 4 already-corrupted Coming Soon rows were corrected back to blank via direct MCP writes.
- **Known-fixed bug (2026-07-23)**: `[W]`'s `EXPECTED:` line could show `⚠ (OVERDUE)` for a Coming Soon game whose expected year was the *current* year — `formatExpected()` (`WatchlistView.jsx`) ran any non-empty `Release Date` string through `new Date(...)` before checking whether that parsed date was in the past. A bare 4-digit-year string like `"2026"` (one of Steam's own valid `release_date.date` display formats, used when Steam only knows the year) silently parses as **January 1st** of that year — which reads as "in the past" the moment the calendar turns past New Year's Day, even though the actual expected window (the rest of that year) hasn't passed. Fixed by detecting a bare-year string first and comparing it against the current year only (same lenient rule the function already used in its year-only fallback branch), never against a fabricated Jan-1 date.

## Completion & Length Tracking

Two more schema additions (added directly via Notion MCP to both the live Games DB and the Starter Template, same presence-gated read/write pattern as the Watchlist properties above): `Completed At` (date) and `Length (hrs)` (number).

- **`Completed At`**: auto-stamped to today the moment a game transitions from non-`Completed` to `Completed` through `App.jsx`'s `handleUpdateStatus` (the Timeline/Random quick-status buttons) — and cleared the moment it transitions back out. Same "only on an *observed* transition" rule as `Released At`: never stamped on a direct add, an Editor save of an already-`Completed` game, or anything that doesn't go through that one code path. Editable in the Editor (to backdate or correct it) once the schema patch is present. Feeds the `[S]` `COMPLETION_VELOCITY` panel and the `[A]` `COMPLETED:` filter — both share one window-matching helper, `src/click-deck/lib/completionTracking.js`.
- **`Length (hrs)`**: HowLongToBeat's "Main + Sides" completion time (their internal field is `comp_plus`/"Main + Extra" — same number, friendlier label), falling back to their "Main Story" stat (`comp_main`) when a less-played or newer title has zero submissions for Main + Extra specifically — confirmed live (*Midnight Scenes: Among Graves* has only `comp_main` on HLTB) rather than excluding an otherwise-real match. HLTB has **no official API** — `api/clickdeck-hltb.js` reverse-engineers their real search handshake (GET `/api/bleed/init` for a one-off `{token, hpKey, hpVal}`, then POST `/api/bleed` with those echoed back in headers + body, both from the same request so the IP/UA binding holds), found by inspecting their own JS bundle and verified live against production before shipping. This is the single most fragile piece of the whole feature — every failure is caught and surfaced as a toast rather than thrown, so a broken/blocked scrape degrades to manual entry instead of taking anything else down. The Editor's **FETCH HLTB** button uses `src/click-deck/lib/hltbMatch.js` (title-confidence scoring, a twin of `steamMatch.js`) to pick the best search result, warning on an unconfident guess the same way FETCH STEAM does. `scripts/backfill-hltb.py` is the one-off Notion backfill (see Scripts & Automation above). Feeds the `[S]` `PLAYTIME_ANALYSIS` panel and the `[A]` `LENGTH:` filter, bucketed via `src/click-deck/lib/lengthBuckets.js` (Short <4h / Medium 4-12h / Long 12-25h / Epic 25h+).

## Discount Banner & Price Tracking

- Whenever any `Backlog`-status game has `Discount Percent > 0`, a banner (`🔥 N GAMES ON SALE! [VIEW]`) appears above the Timeline. Clicking it opens `DiscountModal`, showing every discounted game with its original and sale price and a percent-off badge.
- **Dismiss & snooze**: the banner's `[X]` snoozes it for 24 hours (`localStorage` key `cd_discount_snooze_until`, via `src/click-deck/lib/priceTracking.js`), after which it reappears automatically if anything's still discounted.
- **Persistent mode**: a Settings checkbox (`cd_discount_banner_persistent`) makes the banner permanently visible, ignoring the snooze entirely; the dismiss button hides itself in this mode since it would have nothing to do.
- **Safety net**: a "Show Sale Banner Now" button in Settings clears an active snooze immediately (independent of Save/Close), for the case of dismissing it by accident and wanting it back without waiting out the full 24 hours.
- **Price-drop toast**: on every successful `loadGames()`, the current price of each game is compared against the last price seen for it (a small cache in `localStorage`, `cd_last_prices`) via `detectPriceDrops()`. Any drop toasts `PRICE DROP: [title]` — the client-side equivalent of a push notification, since the nightly pricing cron has no way to reach an already-open browser tab. A game's very first sighting is never reported as a drop (nothing to compare against yet).

## Random Game (`[R]`)

A short slot-machine-style roll through the Backlog (`RandomGameModal`), settling on one game with `[RE-ROLL]` to try again and `[BEGIN PLAYTHROUGH]` to move it straight to `Playing` (preserving any rating it already had, rather than stamping a bogus `0`). Weighting is rank-based (the most-favored candidate gets weight *N* for a pool of *N* games, decreasing to 1) via `src/click-deck/lib/randomWeighting.js`, and the active mode shows as a small badge in the roll modal. A Settings dropdown — **Random Pick Weighting** — offers four modes:
- **Favor your taste** (`taste`, the default): blends **tag affinity** (the average rating of your rated games sharing a candidate's tag) with **studio affinity** (the average rating of your rated games from that developer) at roughly a 2:1 tag:studio weight, via `computeTagAffinity()`/`computeStudioAffinity()`. Falls back to a plain uniform roll below 3 rated games (`COLD_START_MIN_RATED`) — a couple of ratings isn't enough signal to pretend to know your taste — and the badge reads "cold start" in that state so it's obvious why weighting looks inert. `RandomGameModal` takes the rated history as a separate `allGames` prop, distinct from the `backlogGames` candidate pool being ranked.
- **Uniform**: equal odds for every Backlog game.
- **Favor oldest backlog entries**: by `createdTime`.
- **Favor cheapest entries**: games with no known price rank as most expensive (least favored) rather than being excluded.

## Game Editor

Click Deck includes a dedicated **Game Editor Modal** for manual collection management.
- Users can manually add unlisted games, edit existing ones, or completely delete/archive entries from their database.
- Includes fields for Title, Cover URL, Developer, Year, Status, Rating, Tags, and a custom Journal entry.
- Supports pasting external image URLs, or clicking **FETCH STEAM** to search Steam's store by the current title and auto-fill *both* the cover URL and the `Steam App ID` from the best-matching result (via `steamMatch.js`'s `findBestSteamMatch`, not Steam's literal first search result — see the Scripts section above for why that distinction matters). Earlier this button only ever saved the cover URL, silently dropping the App ID — meaning a game covered this way would never get pricing or a working Steam link; that's now fixed. If the best available match doesn't clear the confidence bar (a short/generic title coincidentally overlapping an unrelated listing), it's still applied as a best-effort guess but a `⚠ Uncertain Steam match` toast asks for a manual check instead of silently trusting it.
- Star ratings on the Timeline cards are real, keyboard-reachable `<button>` elements with `aria-label`/`aria-pressed`, not unlabeled clickable `<span>`s.

## Notion Backend & Database Initialization

Click Deck diverges from simpler `localStorage` apps by requiring a real Notion backend to store your game entries.
- The app uses a **Bring Your Own (BYO)** token model.
- Users input their `Notion Integration Token` and their databases' IDs in the **Settings Modal** — that's the entire in-app setup surface. There is no in-app "create a database" or "patch this schema" action.
- **Setup happens in Notion, not in the app**: duplicate the **Click Deck — Starter Template** (and, if using the Watchlist, the **Click Deck — Followed Studios — Starter Template**) from the Dev workspace's Starter Templates section, share the copy with your integration, then paste its database ID into Settings. Both templates ship with the complete, current schema (base + pricing + Watchlist properties) baked in, so there's nothing left to patch afterward. The field guide (`public/click-deck-guide.html`) walks through this end to end.
- This design deliberately traded a self-service "click a button to provision a database" flow for a smaller, more honest surface: the in-app buttons that used to create/patch/reseed a database were development scaffolding from before the Starter Templates existed, not something a real user needs day to day — Notion's own "duplicate this page" is the more reliable place to keep schema in sync going forward, since it can't drift the way a set of independent patch buttons could.

### Adding Data & Cover Art

- The app relies on the Notion database for canonical data — once a token + database ID are configured in Settings, every entry is read from and written to Notion. All entries should be added directly via the UI Editor or via the Notion MCP integration.
- Without a Notion connection the app falls back to a **local-storage demo**: the Onboarding Wizard seeds a curated set of ten genre classics (`PIVOT_TITLES` in `src/click-deck/lib/seed-data.js` — *Loom*, *The Curse of Monkey Island*, *Sanitarium*, *Blade Runner*, *Machinarium*, *Broken Age*, *Firewatch*, *Oxenfree*, *The Wolf Among Us*, *Pentiment*, each with a real Steam App ID and a written-in-voice journal entry) so the timeline, analytics and stats views are explorable offline. This demo data is never written to Notion.
- **Cover Art System**: The Notion integration includes full support for high-resolution external image URLs mapped to the `cover` property of each Notion page. This populates the UI with rich game box art.
- **Interactive Ratings**: Users can click the star buttons directly on the Game Cards on the main timeline to immediately update their rating in the Notion database, minimizing friction. This is an optimistic update (see Architecture above) — it reflects in the UI instantly and only reverts if the Notion write actually fails.

### Adding a New Game Requirements

When adding a new game to the database (whether manually or via automation), ensure the following requirements are strictly met:

1.  **Core Metadata**: Always include the **Title**, **Release Year**, and **Developer/Studio**.
2.  **Cover Art**: The cover must be an external URL fetched from Steam using the format: `https://shared.akamai.steamstatic.com/store_item_assets/steam/apps/<APP_ID>/header.jpg`.
3.  **Tags (Strict Limits)**: Each game **MUST** have a minimum of 5 and a maximum of 7 tags to ensure the Analytics View remains rich but uncluttered. Mix broad genres (e.g., `Point & Click`, `Sci-Fi`) with specific vibes (e.g., `Atmospheric`, `Choices Matter`). The in-app Editor now warns (without blocking the save) when an entry falls outside this range, including legacy entries that predate the 7-tag cap. The canonical tag list, `ALL_TAGS` in `src/click-deck/lib/seed-data.js`, is periodically audited against what's actually live in Notion (via a direct SQL query across the whole collection) to catch drift — a tag someone added directly in Notion that never made it back into the code. `src/click-deck/lib/tagDrift.js` guards new additions to `ALL_TAGS` against accidental near-duplicates (e.g. "Point and Click" alongside the existing "Point & Click"), with a regression test asserting the list itself stays duplicate-free. A `Coming Soon`/`Ignored` Watchlist entry is correctly untagged by design (see [Watchlist](#watchlist-w-anteroom) above) — don't mistake it for a data-quality gap during an audit.
4.  **Journal/Notes (Spicy & Flavorful)**: Notes should *never* be dry factual summaries (e.g., "A game where you click on things"). They should be punchy, opinionated, and flavorful mini-reviews that capture the essence, humor, or tragedy of the game. Use the `Settings` menu (⚙ icon) to edit the raw game data if necessary.

### 🎭 Expressive Rich Text Policy (Journal/Notes)
When generating or updating the `Journal/Notes` property in the Notion database, **you must use expressive Rich Text formatting** to highlight, enforce, and create dramatic tension. 
- Use **colors** (e.g., `red`, `orange`, `green`, `purple`, `pink`) combined with **bolding** for strong emotional or thematic words (e.g., *murder*, *masterclass*, *beautiful*, *dread*).
- Use **italics** and `gray` to denote sarcasm, cynicism, or subtle mystery.
- Use **strikethrough** or **underline** for dramatic misdirection, controversy, or fourth-wall-breaking concepts.
- Be deliberate and dynamic. Do not settle for plain text—ensure the reading experience feels vibrant and dramatically punctuated.

5.  **Status & Rating**: Ensure the play status (`Backlog`, `Playing`, `Completed`, `Abandoned`) is accurate, and rate `Completed` games from 1 to 5.

**Editing without losing formatting**: the in-app Editor preserves a game's existing rich-text formatting *as long as the Journal/Notes text itself isn't touched* during the edit — it detects the plain text is unchanged and writes the original formatted segments straight back (`journalRich` round-trips through `mcp-connector.js`'s `mapGameToProperties`). Editing the journal text itself necessarily produces plain text (a `<textarea>` can't author colored/bold spans), and the Editor shows an inline warning in that case rather than silently flattening the formatting. This guards against what was previously a real data-loss bug: saving *any* field on a richly-formatted entry — even just a status change — used to silently strip all its Notion formatting.

### Disconnecting

There is no in-app "factory reset" action. Since the actual game data lives in Notion (not the browser), disconnecting is just clearing the Token/Database ID fields in Settings and saving — the browser's local-demo cache, if any, can be cleared the same way any site's storage is cleared from the browser itself. Nothing about the collection is ever touched from the app side either way.

## Sync Failure Handling

A Notion sync failure (expired token, revoked integration, rate limit) shows a distinct `SYNC FAILED: [message]` banner with a `[RETRY]` button, rather than the app quietly falling back to "0 records found" as if the collection were simply empty — those two situations previously looked identical, which was more alarming than an actual outage needed to be. When there's no already-loaded data to fall back on, the misleading empty-collection message is suppressed entirely rather than shown underneath the error banner.

## PWA & Cabinet Integration

- Registered in `src/apps-registry.js` under the `react-vite` kind.

Live Notion data still requires a connection; offline just means the shell boots instead of a blank screen. JetBrains Mono is loaded from Google Fonts, so the worker adds a runtime cache-first rule for it rather than precaching it.

- A comprehensive standalone HTML guide (`public/click-deck-guide.html`) provides step-by-step instructions for integrating the Notion backend, initializing (and patching) the database schema, and using every view and control in the app — including a full reference table of the Notion schema.
- The guide is linked directly from the Cone of Cold app catalog (`index.html`).
